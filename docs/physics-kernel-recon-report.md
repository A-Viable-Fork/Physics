## 1. The fork, decided

**Recommendation: physics as an EpiStack corpus, seated as a standalone domain-specific kernel repo on the forked substrate. Not a region inside the `epistack` repo, and not a code-free trellis-style kernel.** This is the hybrid the taxonomy itself names: `docs/kernel-taxonomy.md` defines the domain-specific kernel as the base unit ("A physics kernel, a nutrition kernel"), the kernel-workflow-guide states a built six-stage path for exactly this ("Building your own kernel elsewhere... fork the substrate, then run the workflow against the fork"), and Knowledge-Game demonstrates the operational pattern: vendor the upstream kernel pinned by commit and hash (`upstream/lock.json`, verified on every build), never edit it, put everything domain-specific in your own tree.

Three findings drove the decision.

**First, the machinery is not analogous to the trellis, it is the trellis, mechanized.** The mapping is close enough that reimplementation would be re-derivation of a primitive, which EpiStack's own exclusion reservoir forbids for itself. The correspondences, each verified in code, not docs: the sorry ledger is the characterized-gap reading (`kernel/analysis/characterized-gaps.mjs` literally describes `closing_condition` as "the sorry made a first-class field"); the exclusion reservoir is the withdrawal record with typed reinstatement condition, gate-enforced (`WD-UNSATISFIED` in `kernel/gate/gate.mjs`, checked over the restatement closure so a rephrased identity can be caught if linked); "no instance smuggles a conjecture in as a result" is the declared-versus-earned check (`GM-ABOVE` declines any claim declaring above what it earns, and the Knowledge-Game graph screen renders the divergence); the trellis's generate-and-verify split is the protocol's producer-attenuation stack; the multi-AI cross-exam discipline (Gemini plus DeepSeek independently reproducing the ER-MM-001 bracket) is the independence lift with disjoint source footprints, which is exactly what earns `independently-rechecked`.

**Second, the build cost of reuse is bounded and known; the build cost of a fresh kernel is unbounded.** A trellis-to-v3 translator already exists (`build/translate-trellis.mjs`), pure and deterministic, with the terminal-to-grade and edge-kind mapping tables written down and an acceptance oracle. It consumes the case-graph dialect, not the DG trellis dialect, so a new front end must be written (Section 3), but the target format, the grade semantics, and the determinism discipline are settled. A fresh kernel would re-decide all of it.

**Third, the operator's stated goals are mechanical properties, and only the code provides them.** "Standing recomputable by anyone" is `earned-grade.mjs` plus the certificate seal. "No instance re-litigates a settled kill" is a gate decline, not a prose warning an LLM may skim. "Any instance pointed at the repo inherits the same structure" is the one-schema rule plus the check suite in CI. A trellis-style kernel gets these culturally, through prompts and linter rules that each new instance must choose to honor; an EpiStack corpus gets them structurally. The trellis's own history argues for the structural version: SL-12 records a 20-OoM error concealed for six months by an unstated stipulation, which is precisely the class of failure a mandatory assumptions manifest (refused, not flagged, when empty; `validateTransform` in `kernel/compute/transforms.mjs`) exists to catch.

**Why not a region inside `epistack`.** The submission repo is a frozen competition artifact with a governing trellis of its own, its kind table tuned to its three cases, and a stated rule that adding a case never edits the kernel. The physics needs local kinds, a new translator dialect, new checks, and a corpus an order of magnitude larger than the three cases combined (the trellis alone is ~690 KB of dense prose before atomization). It also needs freedom to diverge from the substrate in one or two small places (Section 3). All of that belongs in a fork that pins upstream, not in the upstream.

**The strongest case for the option not picked** (fresh trellis-style kernel, borrowing discipline but not code): the trellis already works, and its hard part is not structure but verification quality, which the gate does not buy. The protocol spec is explicit that semantic attenuation, whether a named checker exists, whether claimed independence is real, is out of scope and community-level; the gate grades grounding structure only. The operator's pipeline is a one-person community, so the community layer that would police checker honesty is the operator himself, and he already polices it in the trellis. Meanwhile the grade lattice is coarser than the contract verdict ladder (Section 2.3), atomizing 690 KB of prose into claims and links is a real authoring cost, and prose is cheap to revise mid-shore in a way a content-addressed graph is not. This case is serious and it shapes the recommendation: the plan below keeps the trellis as the authoring source initially, with the graph generated by translation, so the cost is paid incrementally and the shore workflow is not disrupted on day one. The rebuttal that decides it: the goals in the brief (kills that block, conjectures that cannot masquerade, structure that transfers to any instance) are exactly the properties the code has and prose lacks, and the one-person-community objection cuts the other way too, since a mechanical gate is most valuable precisely when there is no second human to catch the smuggle.

Derived from the trellis, mapped against what EpiStack builds. The kind table is local policy (protocol spec Section 8; `scaffolder/kernel-config.schema.json` declares local kinds as `{kind, ceiling}` pairs), so nothing here requires a substrate change unless flagged.

**The checker-identity question, named rather than buried.** The pipeline's checkers are AI systems the operator orchestrates. The protocol's `independence: distinct-party` is a declared field the gate trusts structurally; whether Gemini-orchestrated-by-Randy is a distinct party from Randy is a semantic-attenuation question the protocol explicitly leaves to the community. Recommendation: model each external system (Gemini DR, DeepSeek, ChatGPT arbitration) as its own `checker_id` with its own source footprint, declare `distinct-party` for cross-system checks and `self` for same-pipeline Claude checks, and state this convention once in the kernel's own governing document so it is a legible policy rather than an implicit inflation. The trellis already lives this discipline (LINT-19 strips AI narrative, keeps verified computation); the kernel makes it typed.

**The source-class question, surfaced as the workflow guide requires.** The substrate source-class menu (`primary-measurement`, `peer-reviewed`, `preprint`, `dataset`, `institutional-report`, `testimony`) has no honest class for an AI adversarial-audit report. This is a fixed-tier question (the menu is substrate-inherited), and it is the same question Prompt A of the math kernel hit and deliberately stopped on. Options: label audits `testimony` (honest about epistemic status, faintly insulting), or fork the menu with one added class (`ai-audit` or `adversarial-audit`), a one-line, hash-tracked substrate divergence. Recommend the fork, recorded in the lock as a deliberate departure. Judgment call; the build instance may reverse it.

## 3. The seams

**Clean fits, verified in code:** characterized gaps for the sorry ledger; withdrawal-plus-reinstatement for point kills; declared-versus-earned for conjecture smuggling; contradicts-plus-crux for the Path A/B fork; the comment kind for speculation quarantine; append-only store with supersession for shore history (the v4.14 CC-1 statement superseded by the v4.16 annotation is a supersession record, and `WF-SUPERSEDED` then declines any new claim citing the stale version, which mechanizes the S-E1-AUDIT class of problem: stale-citation errors become gate declines instead of audit sorries); the transformation store's "a computation produces a value, never standing" for the trellis's generate-and-verify split; footprint closure for AI-report provenance.

**Strains, each with what would have to be built:**

1. **Class Blocks.** The gate blocks re-entry by identity and restatement closure. A Class Block ("any mechanism relying on local algebraic canonical transformations to achieve ultra-locality") is a predicate over future claims; a rephrased variant gets a fresh identity and passes the gate. This is the largest genuine gap and it cannot be closed inside the protocol without changing what the gate is (content-predicate matching would make the gate non-deterministic in exactly the way the design refuses). Build instead: `build/check-reservoir.mjs`, a physics-kernel check that holds the block claims and their scope patterns and flags candidate violations for the operator, the same relationship the trellis linter already has to shores; plus a reservoir screen in the client (Section 5) so a proposer sees the blocks before submitting. Enforcement is point-mechanical, class-assisted. Honest, and strictly better than the current prose reservoir.

2. **The DG-dialect translator.** `translate-trellis.mjs` consumes the case-graph dialect (nodes with children, composition types, guards, terminals). The DG trellis is structured prose: tiers, K-constraints with bracketed provenance, contract tables, ER entries with typed fields in running text, a sorry ledger with status lines. A new translator front end must parse this dialect (or, more realistically, the trellis-builder conventions get a machine-readable sidecar going forward and the v4.16 corpus is atomized semi-manually with LLM assistance under generate-and-verify). Budget this as the single largest work item. What is settled: the target records, the determinism requirement (two runs byte-identical), the divergence-report-not-paper-over rule, and the acceptance-oracle pattern.

3. **The compute pack taxonomy.** `PACKS` is fixed at `[graph, algebra, statistics]` in the shared validator. Physics numeric transforms (heat-kernel traces, exhaust races, OoM deficits) fit the statistics shape mechanically (`consumes: values`, `emits: value`, never a grade) but "statistics" mislabels them. Either register them under `statistics` with `physics.*` ids (zero divergence, semantic strain) or add a `computation` pack name in the fork (one line, tracked). Recommend the fork, same lock discipline as the source class. Either way, the mandatory assumptions manifest is the payoff: every physics transform declares its regularization scheme, its frame, its independence assumptions, refused if empty, which is LINT-11 and LINT-20 made mechanical. And the run-time guard pattern (`assertNoGradeFields`) extends directly to a convexity guard implementing LINT-CONVEX-1: a matter-channel transform whose output stiffens geometry throws.

4. **Verdict fidelity.** Covered in Section 2: the six-step contract ladder compresses onto the five-position line plus a domain verdict in extensions. Accept the compression; the periphery renders the verdict, the gate never reads it.

5. **Scale, untested.** The built cases are ~60 claims; the physics corpus will be thousands. The grounding computation is memoized linear (V+E, a grounded math-kernel claim), so the algorithm scales, but nothing in the repo has exercised a corpus this size, and the standalone-open constraint (everything inlined into one HTML file) will not survive a corpus of this weight. The physics kernel should drop that constraint (it is a competition constraint of the parent, not a protocol invariant) and serve snapshots the KG way. The build instance learns the real numbers by trying.

6. **Linter rules are procedural.** Only part of the trellis linter compiles to checks. The rest lives as declaration claims and as the kernel's CLAUDE.md-equivalent, which is the honest status: they were always instructions to instances, and they remain that, now versioned next to the graph they govern.

## 4. The Lean decision

**Recommendation: Lean is the constitutive mode of the existing grade lattice, not the gate's closing criterion.** A derivation grounds at the settled-empirical tier through derivation-audit checking records (one distinct-party audit reaches `checked`, two with disjoint footprints reach `independently-rechecked`), and the Lean proof is the `closing_condition` on the residual gap between audited and proved. When a proof lands and compiles, a fork retypes the claim to the `theorem` kind at the constitutive floor. The lattice's mode incomparability (settled-empirical versus constitutive are incomparable by design, and comparing them is a decline) already encodes the true epistemics: an audited derivation and a formal proof are different kinds of settled, not ranks of one kind.

The precedent is in the repo: the math kernel grounds its lattice laws at the constitutive floor by exhaustion (a real proof) and holds its recurrence properties at `checked` by differential testing, with the formal-proof lift recorded as characterized gap G-D. That is exactly the physics situation at larger scale.

The tradeoff, stated: making Lean the gate criterion ("grounds" = "compiles") would buy maximal rigor on the handful of claims Appendix J shows are near formalizable, at the price of flooring essentially the entire corpus at `asserted`, erasing the audit-tier distinctions the trellis spent eighteen months earning, and inverting the incentive (an instance would avoid opening derivation claims at all). The current Lean state seals it: the modules compile but carry structural placeholders (KaluzaKleinExhaust is an R -> R constant with no physical content), a rank-44 proof estimated at hundreds of manual calc lines, and a "decade-scale" tag on the exhaust PDE. A gate that demands what the corpus cannot yet supply produces either an empty kernel or dishonest grades. The chosen design instead makes formalization pull-driven: every unproved derivation permanently displays its distance from the proof floor, and the characterized-gaps read is the formalization backlog, prioritized.

One refinement worth carrying: Appendix J's own module inventory and sorry roadmap should enter the kernel as claims about the formalization state, so "DiracConstraints.lean compiles with the DOF count as arithmetic tautology" is itself a graded, supersedable node rather than an appendix table.

## 5. The client decision

**Recommendation: the Knowledge-Game app, unforked, plus a physics community snapshot, a physics skin, and one new screen shipped through the extension path.** What the physics needs that KG already has, verified in `periphery/`: declared-versus-earned divergence flagging on every graph node (`graph-screen.js` renders it as a first-class visual state); characterized-gap terminals rendered as withheld-record leaves; crux candidates rendered as frontier, never verdict (the Path A/B fork renders correctly with zero new code); support-cone graph layout for derivation dependency structure; the compute picker rendering each transformation's assumptions manifest before anything runs (`compute-screen.js`), with transform-on-graph (recompute-grade, project) already wired; glossary-driven labels resolved from the vendored kernel glossary; and the extension system, hash-installed and conformance-fuzzed so an extension that could move a grade does not load.

What is missing, in order of need:

1. **A reservoir surface.** Withdrawn records exist in store state and the decay read consumes them, but no screen surfaces the kill list as a browsable, searchable object, and the contribute screen does not pre-check a draft against withdrawals and block claims before submit. For the physics, kills being visible and un-re-proposable is half the point. Build: a reservoir screen (kills with their killed-by, their evidence cone, their reinstatement conditions) and a contribute-time advisory check. Ship as an extension plus, for the advisory check, a physics-kernel API read.
2. **Math rendering.** Physics statements are LaTeX-dense; KG cards render plain text. KaTeX is already vendored in the epistack repo (`vendor/katex`), so the precedent and the offline-safe dependency exist; a physics skin adds statement rendering. The skins system (`skin-picker`, `skin-conformance`) is the built path.
3. **Tier semantics in layout.** The support cone is generic; a physics skin should band the layout by the trellis tiers (axioms at the base, K-constraints, functions, contracts, predictions at the frontier) so the inherited structure reads at a glance. Layout is periphery, freely skinnable.

The tradeoff: forking KG would allow deeper surgery (a purpose-built derivation browser) at the cost of maintaining a client, losing the unprivileged-client property KG proves in CI, and forfeiting upstream improvements. The extension path was built precisely so a domain does not fork the client; take it, and revisit only if the skin ceiling is hit in practice.

## 6. The staged plan for the build instance

The trellis stays the authoring source through Stage 3; whether authorship flips to the graph afterward is a decision the build instance makes with real usage data, not one this recon can settle (my lean: it should flip, because instance-inheritance is the goal and a generated trellis view is cheap once the graph is canonical, but the shore workflow's viscosity in graph form is unknowable until tried).

**Stage 0, the seat.** New repo (`physics-kernel` or `dg`). Vendor the epistack substrate pinned by commit and content hash, the `upstream/lock.json` pattern from Knowledge-Game, verified by a check on every build. Record the two deliberate substrate divergences if taken (the `ai-audit` source class, the `computation` pack name) in the lock with rationale. CI runs the vendored check suite plus the physics checks from day one.

**Stage 1, the kernel root.** Scaffolder config: kernel id, adopted shared kinds (measurement, observation, forum, declaration, comment), local kinds per the Section 2 table with their ceilings, and the source table seeded from Appendix E's corpus index with `rests_on` chains (arXiv references, the AI report lineage). Generate the empty kernel. Enter Tier 0 as axiom declarations, Tier 1-2 as standard-result claims with their literature citations and checking records, Tier 3 as adopted conjectures. Commit per stage; the history is the record.

**Stage 2, DG bare.** Enter the DG region floored: CC-1 (as the v4.16 cumulant-form statement, with the v4.14 form entered and immediately superseded so the history is honest), K1-K17, the structural proofs, the live functions, the surviving contracts, both branch commitments with their contradicts link, and every live sorry as a claim with its closing condition. No supports yet; everything earns `asserted`. This commit is the inheritance floor: from here, any instance sees the full structure and cannot mistake an open node for a settled one.

**Stage 3, ground and kill.** Three parallel tracks. (a) Supports: attach the dependency structure per the trellis provenance brackets, conjunctive groups for co-necessary bases, per-child groups for alternatives, `depends-on` for tier commitments and branch gating; attach derivation-audit checking records with per-system checker ids and footprints, so the Gemini-plus-DeepSeek cross-exams earn their `independently-rechecked`. (b) The reservoir: for each of the ~60 THE-DEAD rows and ER entries, enter the kill evidence as claims, the killed mechanism, its withdrawal record, and its reinstatement condition restated as entry-at-grade against a named target claim; enter Class Blocks as block claims. This track is where the translator front end (the single largest build item, budget it first) either exists or the work is LLM-assisted manual under generate-and-verify with spot audits. (c) The physics checks: `check-reservoir.mjs` (class-block advisory), `check-convexity` on the transform layer, and a check that every trellis linter rule is either a compiled check or a cited declaration, so nothing silently drops.

**Stage 4, compute.** Port the load-bearing calculations as transformations with full assumptions manifests: the pole-condition arithmetic, the exhaust race, the OoM-deficit calculator that killed S-EXHAUST-COHERENCE's predecessors, the species-scale conversions (with the E-1 corrected 5D values baked in, so the stale-4D error class is structurally unproducible). Each transform's output lands as a computation claim through the gate, never directly; the register's no-standing rule enforces it.

**Stage 5, the client.** Publish the community snapshot; point KG at it; ship the physics skin (KaTeX statements, tier-banded layout) and the reservoir screen through the extension path. Acceptance: an instance holding only the app can answer "why is this mechanism dead and what would revive it" without reading the trellis.

**Stage 6, flip or hold.** Run one real shore against the graph (the natural candidate is S-CUMULANT-DERIVE, the corpus's own highest-priority gate). If the shore workflow survives graph-native authoring, flip canonicity and generate the trellis as a view; if it fights, hold the trellis as source and keep translating. Decide on evidence.

## Bounds

Confident, verified against code: the mapping table's "clean" rows; the gate, grounding, withdrawal, and compute semantics as described; the translator's existence and dialect gap; the KG capabilities listed; the checks running green. Judgment calls, reversible: the source-class and pack-name forks; comment-kind versus a local speculation kind; the checker-identity convention; extension-path over KG fork. Known unknowns the build instance resolves by trying: corpus-scale performance and snapshot serving; the true authoring cost of atomization and whether the translator front end is worth building versus LLM-assisted manual entry; whether graph-native shores are workable (Stage 6); whether the reinstatement-condition restatement holds up across all ~60 kills rather than the sample I checked. Nothing in the recommendation depends on an unverified claim about the infrastructure; where the brief's description and the repo differed, the repo's version is what is reported (notably: the brief's "canonical (graph, algebra) packs and forkable statistics packs" is accurate, but the brief's framing of the exclusion reservoir as a first-class EpiStack concept overstates it slightly, since the epistack repo's own `docs/exclusion-reservoir.md` is a prose record about the repo's design process, and the mechanical kill-block lives in the withdrawal machinery, which no existing corpus yet exercises in anger; the physics kernel would be its first heavy user, and should expect to find edges).
